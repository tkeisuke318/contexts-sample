# 設計フェーズ — エンハンス

## 規約

- **レイヤー依存方向**：Controller → Service → Repository のみ許可（逆方向・横断禁止）

## 禁止事項

- 要件一覧に存在しない機能を設計書に追加してはならない
- マスタデータになるべき値（区分・コードなど）を画面に直接ハードコードしてはならない
- 外部システム連携設計書なしに外部システム連携の仕様を画面設計書・テーブル定義書に記述してはならない
- Controllerにビジネスロジックを記述してはならない
- Repositoryにビジネスロジックを記述してはならない
- SQLをService層に直接記述してはならない

## レビュー観点マーカー

更新箇所は変更内容がわかるようにインラインで記述する。マーカーなしの記述は「変更前から変わっていない」とみなす。

| マーカー | 意味 |
|---------|------|
| `※変更` | 既存記述を変更した箇所 |
| `※追加` | 新たに追記した箇所 |
| `※削除` | 削除した項目（削除理由を添える） |
| `※推測` | 変更要求に明示されていないが文脈から推論した |
| `※要確認` | 情報不足で判断できない。仮置きしている |

```
# Good
テーブル名: order_details ※変更（旧: order_items）
配送先住所: varchar(200) ※追加
受注区分: — ※削除（コード定義一覧へ移管）

# Bad
テーブル名: order_details
（変更前が何だったかわからない）
```

## 作業手順補足

### Step 1 補足

変更要求・追加機能・修正仕様は `docs/01_要件定義/` 配下から読み込んで変更スコープを確定する。

### 読み込み元

変更スコープに関係する設計成果物を `docs/02_設計/` から読み込む。

### 影響範囲の観点

`rules/02_設計/artifacts.md` の成果物一覧（順序番号・`↑成果物名`）を参照し、波及する成果物を特定する。

- **同フェーズ内**: 例）テーブル定義を変更 → クラス設計書・API設計書・CRUD図も更新対象
- **下流フェーズ**: `docs/03_実装/` 以降の成果物に影響しないか確認する

波及先が多い場合は、更新順序と対象一覧をユーザーに提示して確認を得てから進む。

### Step 4 補足

エンハンスで**新規に追加**する成果物（新画面・新テーブルなど）がある場合のみ、該当テンプレートを `templates/02_設計/` から読み込んで作成する。
