# 設計フェーズ（エンハンス開発）— ノウハウ・暗黙知

## ノウハウ・暗黙知

- API の後方互換性を壊す変更（フィールド削除・型変更）はクライアント側の影響が大きい。変更前に連携先システムを必ず確認する
- DB スキーマ変更（カラム追加・削除・型変更）は既存データとのマイグレーションを伴う。設計段階でマイグレーション方針を決める
- 変更箇所が少なくても、それを呼び出している処理・関連するシーケンス図が多い場合は影響範囲が広くなる
- 設計書の更新漏れは実装フェーズでの問い合わせ多発につながる。変更影響設計一覧を必ず作成してから各設計書を更新する
- 画面項目の追加・削除は画面設計書だけでなく API・バッチ・テーブルへの波及を確認する
