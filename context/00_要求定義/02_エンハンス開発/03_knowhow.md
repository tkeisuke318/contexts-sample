# 要求定義フェーズ（エンハンス開発）— ノウハウ・暗黙知

## ノウハウ・暗黙知

- 変更要求の背景にある業務課題を確認しないまま要求を確定すると、解決策がズレる。「なぜ変えたいのか」を必ず掘り下げる
- 既存の要求定義書が陳腐化している場合がある。変更作業の前に既存成果物と現行システムの一致を確認する
- 変更スコープが曖昧なまま進むと「ついでに直して」が積み重なる。今回変更しないことを明示的に合意する
- 既存要求との矛盾は要件定義フェーズで発覚すると手戻りが大きい。要求定義段階で必ず整合性チェックを行う
- 差分マーカーは成果物の変更履歴管理に直結する。細かすぎる差分よりも「ビジネス的に意味のある変更単位」でマーキングする
