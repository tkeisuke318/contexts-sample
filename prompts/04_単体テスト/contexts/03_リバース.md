# 単体テストフェーズ（リバース）— 作業プロンプト

リバースエンジニアリングとは、既存の実装（ソースコード）からテストケースを生成する作業である。インプットが「設計書・要件」ではなく「実装の事実」である点が新規開発・エンハンスと根本的に異なる。

## Step 1: 生成対象の確定

以下を読み込み、生成するテスト成果物を確定する。

```
contexts/04_単体テスト/01_artifacts.md
contexts/04_単体テスト/02_knowhow.md
contexts/04_単体テスト/03_rules.md
```

生成対象が指示されていない場合は、ユーザーに確認してから次へ進む。

## Step 2: 実装の読み込み

対象成果物に対応する実装を読み込む。読み込む対象はユーザーと合意した範囲に限定する。

| 成果物の種別 | 主な読み込み元 |
|------------|-------------|
| 単体テスト計画書 | クラス一覧・設計書（`docs/02_設計/`）、ソースコード構成 |
| 単体テスト観点表 | ソースコード（対象クラスのメソッド・分岐・例外処理） |
| 単体テストケース | ソースコード（Controller・Service・Repository の実装） |

## Step 3: テンプレートの読み込みと生成

`contexts/04_単体テスト/01_artifacts.md` に記載されたテンプレートパスのファイルを読み込み、実装から読み取った情報でテンプレート構造を埋めて成果物を生成する。

### 推測・補完のルール

リバースは本質的に「実装が正」であるため、マーカーの意味が新規開発と異なる。

| マーカー | 意味 |
|---------|------|
| `※推測` | 実装から読み取れたが意図が不明確。別の解釈の余地がある |
| `※要確認` | 実装を読んでも判断できない。ユーザーへの確認が必須 |
| `※実装と乖離の可能性` | 古いコードや複数箇所に矛盾があり、事実確認が必要 |

```
# Good
テスト観点: nullを渡した場合にIllegalArgumentExceptionをスローする ※推測（コード上は検証しているが、意図的な仕様か防御的コードか不明）

# Bad
テスト観点: nullを渡した場合にIllegalArgumentExceptionをスローする
（なぜそう判断したかわからない）
```

マーカーなしの記述は「実装から明確に読み取れた事実」とみなされる。

## Step 4: 完了報告

生成後、以下をユーザーに報告する。

1. 出力ファイルパス
2. `※推測` の箇所一覧（確認推奨）
3. `※要確認` と `※実装と乖離の可能性` の箇所一覧（確認必須）
