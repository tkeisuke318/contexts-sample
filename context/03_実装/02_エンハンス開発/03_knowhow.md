# 実装フェーズ（エンハンス開発）— ノウハウ・暗黙知

## ノウハウ・暗黙知

- 既存コードを変更する前に、そのコードが他のどこから呼ばれているかを必ず確認する
- リファクタリングと機能変更を同一コミットに混ぜると、変更意図が不明確になりレビューが困難になる
- 既存テストがパスしていることはリグレッションの最低限の確認。テストカバレッジが低い箇所は手動確認を加える
- ブランチ名・コミットメッセージには変更要求チケット ID を必ず含める（追跡可能性の確保）
- 後方互換性を保つために古いフィールド・エンドポイントを残す場合は、deprecated マーキングを付けて期限を明示する
