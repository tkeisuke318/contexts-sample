# コンテキスト設計コンセプト

## 1. 基本概念の定義

| 概念 | 定義 |
|------|------|
| **プロンプト** | LLMへの指示・依頼。LLMに渡されるすべての情報の総称でもある |
| **ドキュメント** | 業務をシステム化するために必要な情報一式。機能仕様・振る舞い・実行環境など静的な情報を記述する |
| **ソースコード** | コンピューターを実際に動かすもの。ドキュメントの最終的な落とし先 |
| **コンテキスト** | AIエージェントが成果物を作るための作業指示。インプット・アウトプット・手順・制約を定義する |

これらは綺麗に区分けして設計しなければならないものではない。ドキュメントもソースコードも、最終的にはすべてLLMに渡されるプロンプト内の情報であり、重なりは許容される。

---

## 2. コンテキストの役割

コンテキストの役割は**作業手順を示す**こと。人がこれまでやってきた作業（5W1H）をAIエージェントに移譲するための記述である。

### AIエージェントが読むことを前提に設計する

コンテキストは主にAIエージェントが読む。ただし人も参照するため、意図が伝わる記述を心がける。AIエージェント向けには以下を原則とする。

- **成果物の生成に直結する情報だけを書く** — 人の役割分担・承認フロー・タイミングは不要
- **条件・制約は明示的に書く** — 「〜を目指す」ではなく「〜であること」
- **インプットとアウトプットを明確にする** — 何を読んで何を作るかが最重要

## 3. CLAUDE.md との違い

CLAUDE.md はシステム全体の概要（アーキテクチャ・技術スタック・コーディング規約）を記述し、**常に全文がシステムプロンプトに埋め込まれる**。

これは2つの問題を引き起こす。

1. **コンテキストウィンドウの圧迫** — すべての会話でフル内容が消費される
2. **ノイズによる精度低下** — 実行中のタスクと無関係な情報が混入し、AIの判断精度に影響する

コンテキストはこの問題を GraphRAG で解決する。**作業手順プロンプト**（`prompts/` 配下）をGraphRAGノードとして管理し、実行タスクに関連するプロンプトだけをグラフ探索で取得してシステムプロンプトに動的注入する。無関係な情報は読み込まれない。

注入されるのはあくまで**作業手順**であり、コンテキスト本体（`contexts/` 配下の成果物定義・ノウハウ・規約）はGraphRAGには含まれない。AIは注入された手順プロンプトの指示に従い、必要な `contexts/` ファイルを明示的に読みに行く。

```
[GraphRAG注入]              [明示的な読み込み]
prompts/02_設計/        →   contexts/02_設計/01_artifacts.md
  contexts/                 contexts/02_設計/02_knowhow.md
    01_新規開発.md            contexts/02_設計/03_rules.md
    02_エンハンス.md          templates/02_設計/...
    03_リバース.md            docs/01_要件定義/...
```

| | CLAUDE.md | コンテキスト（prompts/） |
|---|---|---|
| 内容 | アーキテクチャ・技術スタック・コーディング規約 | 作業手順・contextsの読み方・制約 |
| 粒度 | システム概要レベル | タスク・役割レベル |
| 注入方式 | 常に全文をシステムプロンプトに埋め込む | GraphRAGで関連ノードのみを動的注入 |
| コンテキスト消費 | 全会話で固定コストが発生 | タスクに必要な分だけ消費 |
| ノイズ | タスクによっては不要な情報が混入 | 関連情報のみのため低ノイズ |
