# コンテキスト設計コンセプト

## 1. 基本概念の定義

| 概念 | 定義 |
|------|------|
| **プロンプト** | LLMへの指示・依頼。LLMに渡されるすべての情報の総称でもある |
| **ドキュメント** | 業務をシステム化するために必要な情報一式。機能仕様・振る舞い・実行環境など静的な情報を記述する |
| **ソースコード** | コンピューターを実際に動かすもの。ドキュメントの最終的な落とし先 |
| **コンテキスト** | AIエージェントが成果物を作るための作業指示。インプット・アウトプット・手順・制約を定義する |

これらは綺麗に区分けして設計しなければならないものではない。ドキュメントもソースコードも、最終的にはすべてLLMに渡されるプロンプト内の情報であり、重なりは許容される。

---

## 2. コンテキストの役割

コンテキストは次の2つの役割を持つ。

1. **作業手順を示す** — 人がこれまでやってきた作業（5W1H）をAIエージェントに移譲するための記述
2. **専用エージェントを構成する** — 特定の成果物を作ることに特化したエージェントの振る舞いを定義する

### AIエージェントが読むものとして設計する

コンテキストは人間ではなくAIエージェントが読む。そのため以下を原則とする。

- **成果物の生成に直結する情報だけを書く** — 人の役割分担・承認フロー・タイミングは不要
- **条件・制約は明示的に書く** — 「〜を目指す」ではなく「〜であること」
- **インプットとアウトプットを明確にする** — 何を読んで何を作るかが最重要

## 3. コンテキストが他と異なる点

### データの持ち方とプロンプトへの組み込み方

| | 持ち方 | プロンプトへの組み込み |
|---|---|---|
| ドキュメント / ソースコード | 断片的に分割（従来RAG） | 検索ヒットした断片をユーザープロンプトに追加 |
| **コンテキスト** | **情報間の関係性を保持したグラフ構造（GraphRAG）** | **関連ノードをたどりシステムプロンプト的に常時注入** |

コンテキストの本質は「関係性を持ったデータ構造」と「システムプロンプト的な使われ方」の組み合わせ。

### CLAUDE.mdとの違い

| | CLAUDE.md | コンテキスト |
|---|---|---|
| 内容 | アーキテクチャ・技術スタック・コーディング規約（≈200行） | 成果物の作り方・インプット・制約 |
| 粒度 | システム概要レベル | タスク・役割レベル |
| 検索 | 常に全文が読まれる | GraphRAGで関連するものだけが読まれる |

### Anthropic Sub Agents / Skills との違い

Sub Agentsはコンテキストに最も近い概念だが、設計思想が異なる。

| 観点 | Sub Agents / Skills | コンテキスト |
|------|---------------------|------------|
| **データ構造** | フラットなMarkdownファイル（静的テキスト） | 情報間の関係性を保持したグラフ構造 |
| **検索・取得** | descriptionのテキストマッチング | GraphRAGによるグラフ探索 |
| **注入タイミング** | 呼び出された時だけ | 関連ノードが動的にシステムプロンプトへ注入 |
| **スケーラビリティ** | ファイル数増加で管理が破綻しやすい | グラフ構造なので関係性で整理・拡張できる |

**選択（Sub Agents）vs 自律的収集（コンテキスト）** という設計思想の違い。

## 4. コンテキストファイルの構成パターン

各フェーズのコンテキストは `context/` フォルダ配下の6ファイルに分割する。各ファイルはYAMLフロントマター（`phase` / `kind`）を持ち、GraphRAGの**型付きノード**として機能する。

| ファイル | `kind` | 内容 |
|---------|--------|------|
| `inout.md` | `inout` | インプット（参照ファイル・入手先）とアウトプット（成果物一覧・使用インプット）を定義する |
| `workflow.md` | `workflow` | 成果物を作るステップのみ記述する（レビュー・承認フローは含まない） |
| `knowhow.md` | `knowhow` | 失敗パターン・設計判断の背景など、暗黙知・ノウハウを記述する |
| `rules.md` | `rules` | 命名規則・採番ルール・構造ルールなど成果物の内容に直接影響する規約を記述する |
| `prohibit.md` | `prohibit` | 書いてはならないこと・やってはならない実装パターンを記述する |
| `checklist.md` | `checklist` | フェーズ完了の定義（DoD）をチェックリスト形式で記述する |

各ファイルが独立したノードであるため、GraphRAGは必要な観点（手順だけ、制約だけ、など）を選択的に取得できる。

> `inout.md` のアウトプット表には **`使用インプット` 列**を設け、どのインプットからどの成果物が生まれるかのエッジ情報を明示する。フェーズ内で先行成果物に依存する場合は `↑成果物名` で表記する。

## 5. テンプレートの設計方針

コンテキストが参照するテンプレートには以下の規約を適用する。

- **`★★` 記法** — AIが埋めるべき入力項目の先頭に付与する。各テンプレートの冒頭に記法の説明を記載する
- **図はMermaid形式** — フロー図・ER図・クラス図・シーケンス図はすべてMermaidで記述する

## 6. 具体的なフォルダ構成（システム開発への適用例）

リポジトリルートに `context/`, `docs/`, `templates/`, `old_docs/` を置き、各フォルダの直下にフェーズフォルダを並べる。

```
{リポジトリルート}/
├── context/           # フェーズごとのコンテキスト定義（AIエージェント向け作業指示）
│   ├── 00_要求定義/
│   ├── 01_要件定義/
│   ├── 02_設計/
│   └── ...
├── docs/              # 成果物の格納先
│   ├── 00_要求定義/
│   ├── 01_要件定義/
│   ├── 02_設計/
│   └── ...
├── templates/         # 成果物テンプレート
│   ├── 00_要求定義/
│   ├── 01_要件定義/
│   ├── 02_設計/
│   └── ...
└── old_docs/          # 廃版・旧バージョン
    ├── 00_要求定義/
    └── ...
```

各フォルダの内部構成を設計フェーズを例に示す。

```
context/02_設計/
├── inout.md       # kind: inout  — インプット・アウトプット・使用インプット
├── workflow.md    # kind: workflow — 成果物を作るステップ
├── knowhow.md     # kind: knowhow — 設計判断の背景・失敗パターン
├── rules.md       # kind: rules  — 命名規則・採番・構造ルール
├── prohibit.md    # kind: prohibit — 禁止パターン
└── checklist.md   # kind: checklist — フェーズ完了定義（DoD）

docs/02_設計/
├── 01_画面/       # 画面一覧・画面遷移図・SCR-XXXX_{機能名}.md
├── 02_API/        # API一覧・API-XXXX_{機能名}.md
├── 03_クラス/     # CLS-XXXX_{機能名}.md
├── 04_DB/         # ER図・テーブル一覧・CRUD図・DB-DEF-XXXX_{ドメイン名}.md
├── 05_バッチ/     # バッチ一覧・ジョブスケジュール・BATCH-XXXX_{機能名}.md
├── 06_帳票/       # 帳票一覧・RPT-XXXX_{帳票名}.md
├── 07_コード定義/ # コード定義一覧・CD-XXX_{コード種別名}.md
├── 08_外部連携/   # IF-XXXX_{連携先名}.md
└── 09_システム構成/ # システム全体構成図・サーバー構成一覧 等

templates/02_設計/
├── 01_画面/       # 画面一覧・画面遷移図・SCR-XXXX_{機能名}.md（テンプレート）
├── 02_API/        # ...（docs と同じカテゴリ構成）
└── ...
```

各カテゴリフォルダ内で、プロジェクト全体で1つ作る成果物（一覧・図）はフォルダ直下に、機能・ドメイン単位で複数作る成果物は `ID_{名称}.md` 形式のファイル名で並列配置する。

`inout.md` のアウトプット表には `格納先` 列（例: `docs/02_設計/01_画面/`）を設ける。これにより、AIエージェントは成果物の生成時に格納先とテンプレートの場所を自律的に判断できる。
