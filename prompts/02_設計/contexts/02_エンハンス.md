# 設計フェーズ（エンハンス）— 作業プロンプト

## Step 1: 変更要求の確認

`docs/01_要件定義/` 配下のエンハンス要件（変更要求・追加機能・修正仕様）を読み込み、変更スコープを確定する。

変更スコープが不明な場合はユーザーに確認してから次へ進む。

## Step 2: 既存成果物の読み込み

変更スコープに関係する設計成果物を `docs/02_設計/` から読み込む。

ファイルが存在しない場合は作業を中断し、ユーザーに報告する。

## Step 3: 影響範囲の特定

コンテキストの成果物一覧（順序番号・`↑成果物名`）を参照し、変更が波及する成果物を特定する。

- **同フェーズ内**: 例）テーブル定義を変更 → クラス設計書・API設計書・CRUD図も更新対象になる
- **下流フェーズ**: `docs/03_実装/` 以降の成果物に影響しないか確認する

波及先が多い場合は、更新順序と対象一覧をユーザーに提示して確認を得てから進む。

## Step 4: 更新・生成

特定した成果物を順序に従って更新する。エンハンスで**新規に追加**する成果物（新画面・新テーブルなど）がある場合のみ、該当テンプレートを `templates/02_設計/` から読み込んで作成する。

### 変更箇所の明示ルール

更新した箇所は変更内容がわかるようにインラインで記述する。

| マーカー | 意味 |
|---------|------|
| `※変更` | 既存記述を変更した箇所 |
| `※追加` | 新たに追記した箇所 |
| `※削除` | 削除した項目（削除理由を添える） |
| `※推測` | 変更要求に明示されていないが文脈から推論した |
| `※要確認` | 情報不足で判断できない。仮置きしている |

```
# Good
テーブル名: order_details ※変更（旧: order_items）
配送先住所: varchar(200) ※追加
受注区分: — ※削除（コード定義一覧へ移管）

# Bad
テーブル名: order_details
（変更前が何だったかわからない）
```

マーカーなしの記述は「変更前から変わっていない」とみなされる。

## Step 5: 完了報告

更新後、以下をユーザーに報告する。

1. 更新したファイルパスの一覧
2. `※変更` `※追加` `※削除` の箇所一覧（影響範囲の把握用）
3. `※推測` と `※要確認` の箇所一覧（レビュー要対応）
