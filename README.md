# document-structure

システム開発の各フェーズにおける**ドキュメント構成の定義**と**成果物テンプレート**のサンプルリポジトリです。

[Claude Code](https://docs.anthropic.com/en/docs/claude-code) の Skill を起点に、AIエージェントが各フェーズの成果物を自律的に生成・格納できるよう構成されています。

---

## 使い方

### 基本手順

作業モードに対応する3つのスラッシュコマンドを使います。

```
/newdev  02_設計     ← フェーズの成果物を新規作成する
/enhance 01_要件定義  ← 既存成果物を変更・追加・削除する
/reverse 02_設計     ← 既存実装から成果物を逆生成する（対応フェーズのみ）
```

各コマンドを実行すると、対応する [.claude/skills/](.claude/skills/) 配下の SKILL.md の手順に沿って、`rules/[フェーズ]/` 配下の設定に従って作業が進みます。

成果物ごとの投入用プロンプトは [prompts/](prompts/) 配下にあります。`[画面名]` などのプレースホルダーをプロジェクト固有の情報に書き換えて、スラッシュコマンドに添える形で投入してください。

### 対応フェーズ

| # | フェーズ | 新規開発 | エンハンス | リバース |
|:-:|--------|:-------:|:--------:|:------:|
| 01 | 要件定義 | ✓ | ✓ | |
| 02 | 設計 | ✓ | ✓ | ✓ |
| 03 | 実装 | ✓ | ✓ | |
| 04 | 単体テスト | ✓ | ✓ | ✓ |

リバースは「既存の実装を正」とし、要件定義・実装は対応していません。

## プロジェクト導入時のセットアップ

このリポジトリはサンプルです。プロジェクトで使用する前に以下をテーラリングしてください。

### 1. 成果物の一覧とパスを整える — `rules/[フェーズ]/artifacts.md`

`artifacts.md` には成果物の一覧・格納先パス・テンプレートパス・作成順序・成果物別の記述仕様が記載されています。`docs/` と `templates/` の実構成と常に整合させてください。

- `docs/` の格納先ディレクトリを変更したら `artifacts.md` の格納先を更新する
- `templates/` のパス・ファイル構成を変更したら `artifacts.md` のテンプレート列を更新する
- 成果物を追加・削除したら `artifacts.md` の行・対応テンプレート・格納先ディレクトリを合わせて更新する
- プロジェクト固有の**成果物別の記述仕様**（命名規約・必須カラム・ID採番形式など、値そのものが満たすべき条件）は `artifacts.md` の「成果物別の記述仕様」に追記する

### 2. 規約・禁止事項をプロジェクト標準に合わせる — `rules/[フェーズ]/{newdev,enhance,reverse}.md`

各variantファイルの冒頭に「規約」「禁止事項」セクションがあります。プロジェクトのアーキテクチャ原則・設計ポリシー・禁止事項をここに記述します。

同じ内容を各variantファイルに**重複記載**する構成を採用しています（1ファイル完結性を優先するため）。規約・禁止事項を更新する場合は、同フェーズの `newdev.md` / `enhance.md` / `reverse.md` の該当セクションをすべて揃えて更新してください。

### 3. レビュー観点マーカー・作業手順補足

各 variant ファイルには用途別に以下のセクションも含まれます。プロジェクト固有の事情があれば調整してください。

| セクション | 所在 | 内容 |
|---------|-----|------|
| レビュー観点マーカー | 全 variant | 生成物に付けるインラインマーカー（`※推測`、`※要確認`、`※変更`、`※実装と乖離の可能性` など）の定義。**生成時に付与し、レビュー時に確度・根拠の確認観点として使う** |
| 作業手順補足 | `enhance.md` | 読み込み元・影響範囲の観点など、SKILLの手順を補う情報 |
| 読み込み元 | `reverse.md` | 「成果物 → 実装ファイル」の逆引きマッピング |

> セットアップ後は `.claude/` / `rules/` / `templates/` をプロジェクトリポジトリにコピーし、プロジェクト固有の内容として管理することを推奨します。

## ディレクトリ構成

```
document-structure/
├── .claude/
│   ├── CLAUDE.md                        # プロジェクト共通ルール（セッション開始時に常時ロード）
│   └── skills/                          # バリアント別スキル定義
│       ├── newdev/SKILL.md              # `/newdev {フェーズ}`
│       ├── enhance/SKILL.md             # `/enhance {フェーズ}`
│       └── reverse/SKILL.md             # `/reverse {フェーズ}`
│
├── rules/                               # フェーズ別ルール（各SKILLが読み込む）
│   ├── 01_要件定義/
│   │   ├── artifacts.md                 # 成果物一覧・格納先・テンプレート・記述仕様
│   │   ├── newdev.md                    # 規約・禁止事項・レビュー観点マーカー
│   │   └── enhance.md                   # + 作業手順補足
│   ├── 02_設計/
│   │   ├── artifacts.md
│   │   ├── newdev.md
│   │   ├── enhance.md
│   │   └── reverse.md                   # 規約・禁止事項・レビュー観点マーカー・読み込み元
│   ├── 03_実装/                          # 新規開発・エンハンスのみ
│   └── 04_単体テスト/                    # リバース対応
│
├── templates/                           # 成果物テンプレート（読み取り専用）
│   ├── 01_要件定義/
│   ├── 02_設計/
│   ├── 03_実装/
│   └── 04_単体テスト/
│
├── prompts/                             # ユーザー向け投入プロンプト（variant × 成果物）
│   ├── newdev/                          # 01_要件定義.md / 02_設計.md / 03_実装.md / 04_単体テスト.md
│   ├── enhance/                         # 01_要件定義.md / 02_設計.md / 03_実装.md / 04_単体テスト.md
│   └── reverse/                         # 02_設計.md / 04_単体テスト.md
│
└── docs/                                # 生成された成果物の格納先
    ├── 01_要件定義/
    ├── 02_設計/
    ├── 03_実装/
    └── 04_単体テスト/
```

## ルールファイルの構造

各フェーズの `rules/[フェーズ]/` は以下のファイルで構成されます。

| ファイル | 必須 | 内容 |
|---------|:----:|------|
| `artifacts.md` | ✓ | 成果物一覧・格納先パス・テンプレートパス・作成順序・成果物別の記述仕様 |
| `newdev.md` | ✓ | 新規開発時の規約・禁止事項・レビュー観点マーカー |
| `enhance.md` | ✓ | エンハンス時の規約・禁止事項・レビュー観点マーカー・作業手順補足 |
| `reverse.md` |   | リバース時の規約・禁止事項・レビュー観点マーカー・読み込み元（リバース対応フェーズのみ） |

各 SKILL はコマンド起動時に、対応する variant ファイルを `artifacts.md` とともに読み込みます（例: `/newdev 02_設計` なら `rules/02_設計/artifacts.md` と `rules/02_設計/newdev.md`）。
