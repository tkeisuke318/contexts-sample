# 要件定義フェーズ（エンハンス開発）— ノウハウ・暗黙知

## ノウハウ・暗黙知

- 要件 ID の変更は追跡性を損なう。既存 ID はそのまま維持し、削除要件もマーカーで残す
- 変更要件に紐付くユースケースだけでなく、そのユースケースを参照している他の要件も影響範囲に含める
- 後方互換性の評価は設計・実装フェーズでも行うが、要件定義段階で「互換維持か破壊か」の方針を決めておく
- 非機能要件は変更で知らずに劣化することがある。変更後の性能目標が既存 SLA を下回らないか確認する
- 「ついでにここも直そう」は要件定義段階では受け入れない。変更要求書に含まれていない変更は別チケットにする
