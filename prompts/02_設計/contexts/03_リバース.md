# 設計フェーズ（リバース）— 作業プロンプト

リバースエンジニアリングとは、既存の実装（ソースコード・DB・動作仕様）から設計書を生成する作業である。インプットが「要件」ではなく「実装の事実」である点が新規開発・エンハンスと根本的に異なる。

## Step 1: 生成対象の確定

以下を読み込み、生成する設計成果物を確定する。

```
contexts/02_設計/01_artifacts.md
contexts/02_設計/02_knowhow.md
contexts/02_設計/03_rules.md
```

生成対象が指示されていない場合は、ユーザーに確認してから次へ進む。

## Step 2: 実装の読み込み

対象成果物に対応する実装を読み込む。読み込む対象はユーザーと合意した範囲に限定する。

| 成果物の種別 | 主な読み込み元 |
|------------|-------------|
| 画面一覧・画面遷移図 | フロントエンドのルーティング定義・画面コンポーネント一覧 |
| 画面設計書 | 画面テンプレート・フロントエンドコード |
| ER図・テーブル一覧・テーブル定義書 | DDL・マイグレーションファイル・既存DB |
| API一覧・API設計書 | ルーティング定義・コントローラー実装 |
| クラス設計書・CRUD図 | ソースコード（Controller・Service・Repository） |
| バッチ一覧・バッチ設計書 | バッチ処理ソースコード・ジョブスケジューラ設定 |
| 帳票一覧・帳票設計書 | 帳票出力コード・帳票テンプレートファイル |
| コード定義一覧・コード定義詳細 | Enum定義・定数ファイル・マスタテーブル |
| 外部システム連携一覧・外部システム連携設計書 | 外部連携クライアントコード・設定ファイル |
| システム構成 | インフラ構成ファイル（IaC・サーバー設定・ネットワーク設定） |

## Step 3: テンプレートの読み込みと生成

`contexts/02_設計/01_artifacts.md` に記載されたテンプレートパスのファイルを読み込み、実装から読み取った情報でテンプレート構造を埋めて成果物を生成する。

### 推測・補完のルール

リバースは本質的に「実装が正」であるため、マーカーの意味が新規開発と異なる。

| マーカー | 意味 |
|---------|------|
| `※推測` | 実装から読み取れたが意図が不明確。別の解釈の余地がある |
| `※要確認` | 実装を読んでも判断できない。ユーザーへの確認が必須 |
| `※実装と乖離の可能性` | 古いコードや複数箇所に矛盾があり、事実確認が必要 |

```
# Good
更新日時: updated_at ※推測（カラム名から推測。updated_atが論理削除を兼ねている可能性あり）
承認フロー: 不明 ※要確認（コードに承認ロジックが見当たらない）

# Bad
更新日時: updated_at
（なぜそう判断したかわからない）
```

マーカーなしの記述は「実装から明確に読み取れた事実」とみなされる。

## Step 4: 完了報告

生成後、以下をユーザーに報告する。

1. 出力ファイルパス
2. `※推測` の箇所一覧（確認推奨）
3. `※要確認` と `※実装と乖離の可能性` の箇所一覧（確認必須）
