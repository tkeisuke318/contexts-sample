# document-structure

システム開発の各フェーズにおける**ドキュメント構成の定義**と**成果物テンプレート**のサンプルリポジトリです。

AIエージェントが各フェーズの成果物を自律的に生成・格納できるよう、コンテキスト定義・ドキュメント・テンプレートの置き場所と命名規則を統一しています。

設計コンセプトの詳細は [CONTEXT_CONCEPTS.md](./CONTEXT_CONCEPTS.md) を参照してください。

---

## 使い方

### プロジェクト導入時のセットアップ

このリポジトリはサンプルです。プロジェクトで使用する前に以下をテーラリングしてください。

**1. `contexts/[フェーズ]/01_artifacts.md` と `docs/` / `templates/` をセットで編集する**

`01_artifacts.md` には格納先パスとテンプレートパスが直接記述されています。この3つは常に整合している必要があります。

```
# 01_artifacts.md の1行
| 画面設計書 | docs/02_設計/01_画面/ | templates/02_設計/01_画面/ | ...
               ↑格納先                ↑テンプレートパス
               実際の docs/ 構成      実際の templates/ 構成
               と一致させる           と一致させる
```

- `docs/` の格納先ディレクトリを変更したら `01_artifacts.md` の格納先列を更新する
- `templates/` のパス・ファイル構成を変更したら `01_artifacts.md` のテンプレート列を更新する
- 成果物を追加・削除した場合は `01_artifacts.md` の行・対応テンプレート・格納先ディレクトリをすべて合わせて追加・削除する

**2. `contexts/[フェーズ]/02_knowhow.md` と `03_rules.md` のテーラリング**

ノウハウ・規約はサンプル内容になっています。プロジェクトの実態に合わせて編集してください。

| ファイル | テーラリング内容の例 |
|---------|-----------------|
| `02_knowhow.md` | プロジェクト固有の進め方・設計判断の制約を追記する |
| `03_rules.md` | 命名規則・コーディング規約・禁止事項をプロジェクト標準に合わせる |

> セットアップ後は `contexts/` と `templates/` をプロジェクトリポジトリにコピーし、プロジェクト固有の内容として管理することを推奨します。

### 基本手順

AIエージェントへの指示は2段階で行う。

**Step 1 — コンテキストの投入（学習モード）**

以下の2種類のプロンプトをAIのコンテキストに投入する。これにより、AIはグローバルルールとそのフェーズ・種別での作業手順・制約を把握する。

```
prompts/_global.md                        ← 全フェーズ共通。毎回必ず投入する

prompts/02_設計/contexts/01_新規開発.md   ← 設計・新規開発として作業させたい場合
prompts/02_設計/contexts/02_エンハンス.md  ← 設計・エンハンスとして作業させたい場合
prompts/02_設計/contexts/03_リバース.md   ← 既存実装から設計書を起こす場合
```

**Step 2 — ユーザープロンプトの実行**

`prompts/[フェーズ]/users/` 配下にある成果物ごとのプロンプトを実行する。プロンプト内の `[...]` をプロジェクト固有の情報に書き換えてから投入する。

```
prompts/02_設計/users/画面設計書.md   ← 「[画面名]の画面設計書を作成してください」
prompts/02_設計/users/テーブル定義書.md
```

### contexts プロンプトの種別

| ファイル名 | 種別 | 説明 |
|----------|------|------|
| `01_新規開発.md` | 新規開発 | 要件・設計書をインプットに成果物を新規作成する |
| `02_エンハンス.md` | エンハンス | 既存成果物に変更・追加・削除を加える |
| `03_リバース.md` | リバース | 既存実装（コード・DB・動作仕様）から成果物を生成する（設計・単体テストフェーズのみ） |

## ディレクトリ構成

```
document-structure/
├── contexts/                        # AIがツールで読みに行くフェーズ別定義
│   ├── 00_要求定義/
│   │   ├── 01_artifacts.md          # 成果物一覧・格納先・テンプレートパス
│   │   ├── 02_knowhow.md            # 進め方・ノウハウ・暗黙知
│   │   └── 03_rules.md              # 規約・禁止事項
│   ├── 01_要件定義/                  # 同上（以降同構造）
│   ├── 02_設計/
│   ├── 03_実装/
│   ├── 04_単体テスト/
│   ├── 05_結合テスト/
│   ├── 06_システムテスト/
│   ├── 07_受入テスト/
│   ├── 08_移行設計/
│   └── 09_運用設計/
│
├── prompts/                         # AIへの指示プロンプト（学習モードで投入）
│   ├── _global.md                   # 全フェーズ共通ルール（毎回必ず投入）
│   ├── 00_要求定義/
│   │   ├── contexts/                # 作業手順プロンプト（学習モードで投入）
│   │   │   ├── 01_新規開発.md
│   │   │   └── 02_エンハンス.md
│   │   └── users/                   # ユーザープロンプト（成果物ごとに実行）
│   │       ├── ヒアリングシート.md
│   │       ├── 課題一覧.md
│   │       ├── 要求定義書.md
│   │       └── ビジネスプロセス図.md
│   ├── 01_要件定義/                  # 同上（以降同構造）
│   ├── 02_設計/
│   │   ├── contexts/
│   │   │   ├── 01_新規開発.md
│   │   │   ├── 02_エンハンス.md
│   │   │   └── 03_リバース.md       # 設計フェーズはリバースあり
│   │   └── users/
│   │       ├── 画面一覧.md
│   │       ├── 画面設計書.md
│   │       ├── ER図.md
│   │       └── ...（成果物ごとに1ファイル）
│   ├── 03_実装/
│   ├── 04_単体テスト/
│   │   ├── contexts/
│   │   │   ├── 01_新規開発.md
│   │   │   ├── 02_エンハンス.md
│   │   │   └── 03_リバース.md       # 単体テストフェーズはリバースあり
│   │   └── users/
│   ├── 05_結合テスト/
│   ├── 06_システムテスト/
│   ├── 07_受入テスト/
│   ├── 08_移行設計/
│   └── 09_運用設計/
│
├── templates/                       # 成果物テンプレート
│   ├── 00_要求定義/
│   ├── 01_要件定義/
│   ├── 02_設計/
│   └── ...（フェーズごとに配置）
│
└── docs/                            # 生成された成果物の格納先（gitignore推奨）
    ├── 00_要求定義/
    ├── 01_要件定義/
    └── ...（フェーズごとに配置）
```
